## Rewritten descriptions

### list_tables

List all tables in the SQLite database. Use this ONLY when the user gives no hint about any table name and you cannot infer one from the request. If the user refers to users, orders, products, or similar records, assume the matching table exists and go directly to read_query or write_query; do not call this tool for verification.

### describe_table

Get the schema (columns and types) for a specific table by providing its table_name. Use this ONLY when the user explicitly asks about table structure, or when the request gives absolutely no clue about which columns to use. If the user mentions common fields like a name, email, or total, infer the column names and use read_query or write_query directly instead; never call this tool before those just to verify.

### read_query

Execute a SELECT query to read data from the SQLite database. Use this directly for counts, aggregations, joins, or any data retrieval; never call list_tables or describe_table beforehand to verify tables or columns. If the user mentions entities like users, orders, or products, treat those as known table names and query immediately. Use describe_table only when the user explicitly asks for column names or table structure.

### write_query

Execute an INSERT, UPDATE, or DELETE query to modify data in the SQLite database. Use this directly when the user asks to add, change, or remove rows; never call list_tables or describe_table beforehand to verify the table or columns. Infer standard column names from context (e.g., name, email, total) and execute immediately. To create a new table, use create_table instead.

### create_table

Create a new table by executing a CREATE TABLE query. Use this directly when the user asks to define a new table with specific columns. Do not verify creation afterward with list_tables; to modify existing data, use write_query instead.
