The loop other tools leave open

Eval runners tell you agents misuse your tools. Toolmetry is the step after: fix, re-measure, prove — and throw the fix away if it doesn't measure better.

  1. Measure

    Realistic scenarios run N=5 against your live server. Every actual tool call is recorded — hits, wrong args, extra calls.

  2. Diagnose

    Failing runs plus your current descriptions go to an LLM that explains which description made the wrong tool look right.

  3. Rewrite

    Descriptions are rewritten and applied in-memory — your server is never modified.

  4. Prove

    Re-measure. Improvements are kept; regressions are discarded automatically. Ship winners through toolmetry proxy, no fork needed.

Three ways descriptions fail agents

All three archetypes showed up in the official servers — and all three were fixable with words alone.

Wrong-tool confusion

Sibling tools with overlapping descriptions make agents guess. The memory server's create/add and search/open pairs confused agents once in five runs.

memory: hit rate 80% → 100%

Ritual extra calls

When a description doesn't say "you don't need to check first," agents check first. Every time. One added sentence ended schema-peeking before queries.

sqlite: extra-call rate 66% → 0%

Deprecated-alias traps

A deprecated tool whose description still reads like the primary one is a trap agents walk into forever. Lead with the deprecation, name the replacement.

filesystem: strict success +10 pts

What this can't do

Data without caveats is marketing. Read these before quoting the table.

  • N=5 quantizes. Per-scenario rates move in 20-point steps; trust aggregates, not single scenarios.
  • Rewrites are high-variance. Two attempts on one baseline scored +10.0 and −2.2 points. Measurement, not faith, is the product — the loop discarded that regression automatically.
  • Deltas are agent-specific. A stronger agent (Claude Haiku 4.5) started near the optimized score and gained only +2.2. Bad descriptions tax cheap agents hardest.
  • Scenarios encode the ground truth. A debatable expected tool makes a debatable metric. All scenarios are in the repo — judge them.
  • Words can't fix tool design. Overlapping capabilities or missing parameters survive any rewrite. The report shows which scenarios stayed broken.
